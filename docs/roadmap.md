# Roadmap — что добавить

Список того, чего системе не хватает, отсортированный по фактической ценности.

## Реалистичная оценка текущего состояния

**Что работает на реальных данных:**
- ✅ Каркас фрод-движка (NDVI + SAR backscatter + Coherence + cross-validation + rain-фильтр)
- ✅ NDVI через Sentinel Hub Cloud
- ✅ SAR backscatter через CDSE (182+ точек в БД)
- ✅ Coherence через ASF HyP3 (Earthdata Bearer auth → CDSE catalog → INSAR_GAMMA → geotiff clip)
- ✅ Triple-validation (NDVI + SAR + Coherence) с finding `CROP_TRIPLE_VALIDATED`
- ✅ Кеш в Postgres + дисковый, прогрев при регистрации
- ✅ 19 unit-тестов на синтетических рядах (11 SAR + 8 Coherence)
- ✅ Два UI-пути: F-* (мок) и U-* (реальный юзер)
- ✅ Никаких mock-fallback'ов в продакшен-путях
- ✅ **Per-parcel storage**: каждый участок хозяйства хранится отдельно
  (`UserField.parcels: Parcel[]`) — со своим контуром, агрохимией и кадастр-номером
- ✅ **Lazy spутник**: базовая страница инспектора рендерится без SH/CDSE-запросов;
  каждый parcel показывает «▶ Посмотреть со спутника», на клик → `?sat=fi:pi` →
  инлайн SatelliteSection только для этого parcel'а
- ✅ **Геодезическая площадь** (Бэвис-Камбарелли) + габариты + центроид-координаты
  на каждой parcel-карточке
- ✅ **Coherence-jobs indicator**: бейдж «N в обработке / N готов» на parcel'е
  по данным `field_coherence_jobs`

**Coherence — статус end-to-end (на дату snapshot документации):**
- ✅ Auth, catalog, submit, polling — все подтверждены живыми данными
- 🟡 Финал-фаза (download + clip + write в БД) — код есть, end-to-end не
  прогонялась пока ни один HyP3-джоб не достиг SUCCEEDED. Ожидаемо
  заработает на первом DONE; известные edge-cases (UTM-tile, DEFLATE-zip)
  имеют точечные fix'ы.

**Что не доказано:**
- ❌ Что детекторы реально ловят фрод на казахстанских полях.
  Пороги (-3.5 дБ ΔVH, γ < 0.3, 30 дн. расхождение, и т.д.) пока
  **из литературы** и работают на синтетике. Нужен backtest на реальных
  полях с известной историей событий (см.
  [operations.md](./operations.md#backtest-на-реальных-полях)).

**До тех пор пока backtest не сделан — называть систему «работает» преждевременно.**
Это самое важное, что нужно сделать дальше.

## Приоритет 1 — без этого нельзя доверять числам

### 1.1. Backtest на 5–10 реальных полях

**Зачем:** доказать, что precision/recall детектора > 70 % на реальных
казахстанских данных, а не только на тестах автора.

**Что делать:**
1. Открыть EO Browser, выбрать поля 2024–2025 с явными циклами.
2. Глазами разметить даты sowing / harvest, заполнить
   `scripts/sar-backtest.fixtures.json`.
3. Прогнать `npm run sar:backtest`, получить precision/RMSE.
4. Если precision < 70 % — тюнить пороги.

**Сложность:** 2–4 часа человеческого времени на разметку.
**Блокер:** нужно сделать своими руками — сценарии нельзя сгенерировать.
**Файлы:** [scripts/sar-backtest.ts](../scripts/sar-backtest.ts).

### 1.2. Per-crop тюнинг порогов

**Зачем:** яровая пшеница, озимые, подсолнечник, люцерна — у всех разные
сигнатуры. Один порог `HARVEST_VH_DROP_DB = 3.5` для всех — компромисс.

**Что делать:**
- В `SAR_THRESHOLDS` ([lib/satellite/sar-events.ts](../lib/satellite/sar-events.ts))
  добавить per-crop оверрайды:
  ```ts
  const CROP_OVERRIDES: Record<Crop, Partial<typeof SAR_THRESHOLDS>> = {
    wheat_spring: { HARVEST_MONTH_MIN: 8, HARVEST_MONTH_MAX: 9 },
    sunflower:    { HARVEST_MONTH_MIN: 9, HARVEST_MONTH_MAX: 10 },
    ...
  };
  ```
- В `detectSAREvents` принимать `crop?: Crop` и применять оверрайды.
- Прогнать backtest с разделением по культуре.

**Сложность:** ~3 часа кода + час на тесты.
**Зависит от:** 1.1 (без числовой обратной связи тюнить нечего).

## Приоритет 2 — закрывает реальные дыры

### 2.1. NDVI-фетч в Postgres-кеш

**Зачем:** сейчас NDVI кешируется на диск (`/tmp/agro-sat-cache/`), что
не переживает рестарт контейнера и не работает между процессами. SAR уже
в БД — логично туда же положить и NDVI.

**Что делать:**
- Расширить `field_sar_observations` (или новая таблица) на NDVI-точки.
  Колонка `ndvi` уже есть в схеме, осталось писать.
- Перенести логику кеширования из `lib/satellite/cache.ts` для namespace `ndvi`
  в БД-кеш.

**Сложность:** ~2 часа.
**Файлы:** [lib/satellite/cache.ts](../lib/satellite/cache.ts), [lib/satellite/sentinel-hub-provider.ts](../lib/satellite/sentinel-hub-provider.ts).

### 2.2. Авто-cross-validation для NDVI ↔ SAR событий sowing/tillage

**Зачем:** сейчас cross-validation работает только для `harvest`. Если SAR
говорит «вспашка 25 апреля», а NDVI неделю назад показывал bare soil — это
тоже усиливает confidence. Аналогично с sowing.

**Что делать:**
- В `runSatelliteChecks` ([lib/verify/satellite.ts](../lib/verify/satellite.ts))
  добавить cross-validation для NDVI `growthStartDate` ↔ SAR `sowingEvent`.
- Новый код `CROP_SOWING_CROSS_VALIDATED` для согласованных дат.

**Сложность:** ~1 час.

### 2.3. Уменьшить число запросов к SH/CDSE

**Зачем:** сейчас на странице досье делаются повторные запросы:
1. `verifyFarmerWithSatellite` тащит SAR
2. `SatelliteSection` тащит SAR заново (хотя в БД-кеш уже есть)
3. Для real users — тот же fetch ещё в `RealUserPage` для `sarSummary`

Все попадают в кеш, но это всё равно 3 чтения из БД и 3 lookup-а в Redis.

**Что делать:**
- Сделать `SatelliteContext` в page.tsx, который один раз тащит NDVI/SAR
  и пробрасывает во все консьюмеры через React `cache()` или прямо props.

**Сложность:** ~2 часа рефакторинга.

### 2.4. Tile-based детектор для гетерогенных полей

**Зачем:** Statistical API усредняет VH по всему полигону. Для поля 50 га,
половина которого в работе, а половина под паром, средний VH будет «не
рыба, не мясо». Реальные поля Гипрозема в KZ — несколько участков под одним
именем.

**Что делать:**
- Разбить полигон на сетку тайлов 100×100 м.
- Для каждого тайла отдельный fetch (или один с разбиением через
  `multiOutput`-evalscript).
- Детектор — на каждом тайле, агрегация по тайлам.
- В UI — подсветка тайлов: «вот эта четверть поля точно убрана, эта спит».

**Сложность:** 2–3 дня. Это уже отдельный архитектурный пласт.
**Ценность:** очень высокая для крупных хозяйств.

## Приоритет 3 — операционная зрелость

### 3.1. Системный cron + alerting

**Зачем:** сейчас `/api/satellite/sar/refresh` дёргается вручную или при
регистрации. Для прода нужен:
- Раз в неделю прогревать всех зарегистрированных юзеров.
- Алерт в Slack / на email если refresh упал.

**Что делать:**
- Vercel Cron / k8s CronJob / linux crontab — выбрать под инфру.
- Логировать результат в Postgres (`refresh_log`) или внешний sink.

**Сложность:** ~1 час под выбранную инфру.

### 3.2. Мониторинг Processing Units CDSE

**Зачем:** при 1000+ полей-сезонов можно нечаянно сжечь месячную квоту PU
за день. Сейчас никак не отслеживается.

**Что делать:**
- В CDSE Dashboard смотреть PU вручную — там есть.
- Логировать `Sentinel-Hub-Processing-Units-Spent` из ответа в `field_sar_observations.processing_units` (или отдельная таблица `sh_usage_log`).
- Дашборд: «израсходовано N PU за неделю, осталось ~M».

**Сложность:** ~2 часа. Но без программного API CDSE на PU — это всё равно
оценка из заголовков ответов.

### 3.3. Real users SAR-финдинги в общем risk-score

**Зачем:** сейчас для U-* юзеров SAR-финдинги показываются на карточках
заявок (через `checkUserApplication`), но они не идут в общий `verdict.riskScore`,
который суммирует риск всех нарушений. Для F-* (мок) — идут (через `verifyFarmerWithSatellite`).

**Что делать:**
- В [app/inspector/farmers/[id]/page.tsx](../app/inspector/farmers/%5Bid%5D/page.tsx)
  для `RealUserPage` собрать все SAR-warnings из всех заявок и показать
  агрегированный «Выявленные нарушения · N» сверху.

**Сложность:** ~1 час.

## Приоритет 4 — расширения сигналов

### 4.1. ~~Coherence (CCD)~~ — ✅ **DONE через ASF HyP3**

Изначально планировалось как BYOC или своя SNAP-инфра. По факту реализовано
через **ASF HyP3** (бесплатное NASA-облако): submit SLC-пар через CDSE
catalog, async-обработка, download `_corr.tif`, ray-casting clip mean γ.

**Что осталось дотюнить (точечные fix'ы, не архитектура):**
- proj4js-reproject если HyP3 отдаст tile в UTM (~30 строк)
- pako-inflate если HyP3 решит сжать coherence.tif внутри zip (~10 строк)
- ASF Search для перехода на `INSAR_ISCE_BURST` (быстрее, ~50 строк)
- Раздробить refresh-эндпоинт на PHASE1/PHASE2/PHASE3 чтобы влезть
  в 5-мин лимит Vercel Cron при > 50 полигонах

См. [coherence.md](./coherence.md#известные-edge-cases).

### 4.2. NDWI / NDMI — irrigation detection

**Зачем:** сейчас «субсидия на ирригацию» проверяется только косвенно
(через NDVI). Прямой канал — NDWI (water content) или NDMI (moisture):
полив поднимает NDWI на ~0.1 в течение 3-5 дней.

**Что делать:**
- Добавить evalscript для NDWI в [lib/satellite/sentinel-hub-provider.ts](../lib/satellite/sentinel-hub-provider.ts).
- Детектор «всплески NDWI» — те же приёмы, что в NDVI/SAR.
- Новый код `CROP_IRRIGATION_NOT_DETECTED` для заявок на ирригацию без всплесков.

**Сложность:** ~3 часа.

### 4.3. Обнаружение скота на пастбище через S2 / S1

**Зачем:** скотоводческий блок сейчас полагается только на ИАС/Plem.kz.
Реальное присутствие животных можно ловить через:
- NDVI на пастбище (выпас → стрижка трав → NDVI ниже окружающего фона)
- SAR coherence (топтание → нарушенная структура почвы)

**Что делать:** долгий research-проект. Сначала backtest на 2-3 пастбищах
с известной нагрузкой, потом думать о коде.

**Сложность:** недели.

## Приоритет 5 — UX и наблюдаемость

### 5.1. Inspector dashboard для всех findings

**Зачем:** инспектор сейчас видит фермеров по списку, но нет общего
**списка финдингов** по типу: «все CROP_HARVEST_CROSS_VALIDATED за неделю»,
«все SAR_FIELD_INACTIVE по СКО».

**Что делать:** новая страница `/inspector/findings` с фильтрами по
коду, региону, severity, дате.

**Сложность:** ~4 часа.

### 5.2. Карта полей с подсветкой проблемных

**Зачем:** для региональных инспекций карта KZ с маркерами по риск-скору
ценнее списка.

**Что делать:** Leaflet (уже в зависимостях), отдельная страница `/inspector/map`.

**Сложность:** 1 день.

### 5.3. Email-уведомления фермерам о warning-ах

**Зачем:** сейчас фермер видит фрод-проверку только при заходе в кабинет.
Если автоматика нашла нарушение — он узнаёт об этом, когда заявка отклонена.

**Что делать:**
- При создании заявки прогнать `checkUserApplication` синхронно,
  если есть `high`/`critical` — отправить юзеру письмо: «На вашей заявке
  X сработала автопроверка. Вот что нашлось. Вы можете уточнить
  декларацию до отправки на инспектора».

**Сложность:** ~3 часа + надо интеграцию с SMTP/Resend/SES.

## Что **сознательно** не делаем

- **Удобрение и ирригация по SAR.** Сигналы в backscatter настолько слабые,
  что precision не пробьёт 50 %. Будет много шума и ложных срабатываний.
  Только NDWI или ничего.
- **Real-time детекция (< 1 ч задержки).** S1 + S2 публикуются с лагом 2-4 дня.
  Любой «real-time» здесь — обман.
- **ML-классификация культур по полю.** Это отдельная R&D задача, не наш
  продукт. Используем декларацию + Гипрозем-attribute.
- **Прогноз урожая.** Не наша территория. Мы валидируем заявленные показатели,
  не предсказываем.

## Если бы делал я с нуля сегодня

1. **Backtest сразу.** До любого кода — 10 размеченных полей и метрика, к
   которой стремимся. Сейчас этого нет, и любые «улучшения» детектора
   слепые.
2. **Один кеш-слой.** Сейчас два (диск + БД), и они разошлись по namespace.
   Унификация в БД с миграцией NDVI = меньше ментальной нагрузки.
3. **Все спутниковые операции через одну точку.** Сейчас есть `getNDVITimeseries`,
   `getS1Series`, `getSAREvents`, `verifySatellite`, `checkInactivity` — много
   точек входа, легко перепутать. Один `SatelliteContext` per (полигон, год)
   — было бы чище.
